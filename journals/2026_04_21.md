- ## Desarrollo Técnico
	- **Creación del portal web calaire-app (Etapa 1 — Setup base):** se inició la construcción del portal web del programa [[CALAIRE-EA]] para la gestión de rondas de ensayos de aptitud. El stack tecnológico seleccionado es Next.js 16 (App Router con Server Components y Server Actions) para frontend/backend, WorkOS para autenticación, y Supabase (PostgreSQL) como base de datos. Se diseñó y ejecutó el esquema SQL con 4 tablas relacionales: `rondas` (maestro de rondas), `ronda_contaminantes` (configuración por contaminante con niveles y réplicas), `ronda_participantes` (asignación de laboratorios a rondas) y `envios` (registros de datos de participantes). La aplicación reemplaza el prototipo abandonado que existía en `calaire-ea/apps/portal-rondas-ea`.
	  date:: 2026-04-21
	  category:: [[Desarrollo Técnico]]
	  status:: completed

El sistema de diseño se construyó con CSS puro (variables CSS) e incluye las clases `.card`, `.card-accent`, `.header-bar`, `.btn-primary`, `.btn-outline` y `.numeric`. Esta decisión de diseño garantiza coherencia visual entre el portal web y la aplicación de procesamiento estadístico, fortaleciendo la identidad institucional del programa de ensayos de aptitud.

	- **Dashboard del coordinador (Etapa 2):** se implementó el panel administrativo para la gestión de rondas de ensayo. Las funcionalidades incluyen: creación de rondas con estado inicial de borrador, configuración por contaminante (niveles y réplicas), transiciones de estado del ciclo de vida de la ronda (`borrador → activa → cerrada`), y visualización consolidada de todas las rondas existentes. La implementación utiliza Server Components para renderizado en servidor y Server Actions para mutaciones de datos, eliminando la necesidad de endpoints API REST separados.

- ## Aprendizajes y Decisiones
	- **Decisión de invitación sin correo automático:** se adoptó el modelo de invitación mediante enlace manual con tokens pre-generados. Los coordinadores generan tokens únicos por participante y comparten el enlace de acceso por los canales de comunicación existentes (correo institucional, WhatsApp). Esta decisión simplifica la arquitectura al eliminar la dependencia de servicios de correo transaccional y mantiene el control operativo en manos del coordinador de la [[Prueba Piloto]].
	  date:: 2026-04-21
	  category:: [[Desarrollo Técnico]]
	  status:: documented
	- **Decisión de cierre con submitted_at:** se decidió no crear una tabla separada de submissions. La marca temporal `submitted_at` existente en la tabla `envios` cumple doble función: trazabilidad del momento exacto del envío y mecanismo de bloqueo post-envío. Esta decisión reduce la complejidad del esquema de base de datos sin sacrificar funcionalidad.
